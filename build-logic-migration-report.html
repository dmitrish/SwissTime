Project Overview

Goal: Migrate the SwissTime Android project from individual module build configurations to a centralized convention plugin system for improved maintainability and consistency.

Problems Encountered & Solutions

üö® Problem 1: Version Catalog Conflict

Error: "Invalid catalog definition: In version catalog libs, you can only call the 'from' method a single time"

‚ö†Ô∏è Initial Solution (Flawed)

Initially created a duplicate libs.versions.toml in the build-logic directory. The error occurred because both the main project and build-logic were trying to define the same version catalog.

First attempt: Removed the duplicate version catalog from build-logic and used direct version strings in dependencies instead.

// Initial (problematic) approach: dependencies { compileOnly("com.android.tools.build:gradle:8.10.0") // ‚ùå Hardcoded versions compileOnly("org.jetbrains.kotlin:kotlin-gradle-plugin:2.2.10") // ‚ùå Duplication }

Problem with this approach: This introduced version duplication, defeating the purpose of convention plugins. The same versions existed in both gradle/libs.versions.toml and hardcoded in build-logic dependencies.

‚úÖ Final Solution (Proper)

Root Cause Analysis: The issue was trying to create a separate version catalog for build-logic instead of sharing the main project's catalog.

Proper Resolution: Added build-logic specific dependencies to the main gradle/libs.versions.toml and properly accessed the shared catalog from convention plugins.

gradle/libs.versions.toml
# Build logic dependencies (added to main catalog) android-tools-common = { group = "com.android.tools", name = "common", version = "31.8.1" } android-gradle-plugin = { group = "com.android.tools.build", name = "gradle", version.ref = "agp" } kotlin-gradle-plugin = { group = "org.jetbrains.kotlin", name = "kotlin-gradle-plugin", version.ref = "kotlin" } compose-gradle-plugin = { group = "org.jetbrains.kotlin", name = "compose-compiler-gradle-plugin", version.ref = "kotlin" }
build-logic/convention/build.gradle.kts
dependencies { compileOnly(libs.android.gradle.plugin) // ‚úÖ Uses shared version catalog compileOnly(libs.android.tools.common) // ‚úÖ Single source of truth compileOnly(libs.kotlin.gradle.plugin) // ‚úÖ No duplication compileOnly(libs.compose.gradle.plugin) // ‚úÖ Maintainable }
AndroidCompose.kt
val libs = the<VersionCatalogsExtension>().named("libs") dependencies { add("implementation", platform(libs.findLibrary("androidx.compose.bom").get())) add("androidTestImplementation", platform(libs.findLibrary("androidx.compose.bom").get())) add("debugImplementation", libs.findLibrary("androidx.ui.tooling").get()) }

Key Benefits of This Approach:

  • Single source of truth for all versions
  • No version duplication between main project and build-logic
  • Easy to update AGP, Kotlin, or Compose versions project-wide
  • Consistent with Android development best practices

The KSP (Kotlin Symbol Processing) plugin dependency was not needed for this project's convention plugins since we weren't using KSP for code generation.

üö® Problem 3: Deprecated kotlinOptions API

Error: "Using 'kotlinOptions(KotlinJvmOptionsDeprecated)' is an error. Please migrate to the compilerOptions DSL"

‚úÖ Solution

Kotlin 2.2.10 deprecated the kotlinOptions API in favor of the new compilerOptions DSL for better type safety and consistency.

Resolution: Migrated from the old kotlinOptions syntax to the new compilerOptions DSL:

// Old (deprecated): kotlinOptions { jvmTarget = "11" allWarningsAsErrors = warningsAsErrors.toBoolean() freeCompilerArgs = freeCompilerArgs + listOf(...) } // New (current): compilerOptions { jvmTarget.set(org.jetbrains.kotlin.gradle.dsl.JvmTarget.JVM_11) allWarningsAsErrors.set(warningsAsErrors.toBoolean()) freeCompilerArgs.addAll(listOf(...)) }

üö® Problem 4: Version Catalog Access in Convention Plugins

Error: "Unresolved reference: libs" when trying to access version catalog in AndroidCompose.kt

‚úÖ Solution

Convention plugins run in a different Gradle context and don't automatically have access to the generated libs accessor. Need to explicitly access the version catalog.

Resolution: Use the VersionCatalogsExtension API to access the shared version catalog:

import org.gradle.api.artifacts.VersionCatalogsExtension import org.gradle.kotlin.dsl.the internal fun Project.configureAndroidCompose( commonExtension: CommonExtension<*, *, *, *, *, *>, ) { val libs = the<VersionCatalogsExtension>().named("libs") // ‚Üê Explicit catalog access dependencies { add("implementation", platform(libs.findLibrary("androidx.compose.bom").get())) // ... other dependencies } }

Why this approach works:

  • Provides explicit access to the version catalog from any Project context
  • Works consistently across all convention plugin functions
  • Maintains type safety with proper error handling
  • Follows Gradle's recommended pattern for catalog access in plugins
